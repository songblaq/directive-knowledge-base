# DKB Vision

## Why DKB?

AI 에이전트 도구 생태계가 급속히 확장되면서, agent, skill, plugin, workflow, command 등 다양한 이름으로 불리는 역할 아티팩트가 쏟아지고 있다. 이들은 서로 다른 도구(Claude Code, Codex, OpenCode, Cursor, VS Code 등)에서 각기 다른 형태로 제공되지만, 본질적으로는 **"AI에게 특정 역할과 행동을 지시하는 프롬프트"**라는 공통점을 가진다.

문제는 이것들을 그냥 설치하면 생긴다:
- "plan" 스킬이 3개, "review" 스킬이 4개 — 뭐가 뭔지 모른다
- 설명도 부실하고, 실제 동작도 이름과 다를 수 있다
- 공식인지 비공식인지, 안전한지 아닌지 판단할 근거가 없다
- 중복되는 것, 겹치는 것, 충돌하는 것을 걸러낼 방법이 없다

## What DKB Does

DKB(Directive Knowledge Base)는 이 문제를 체계적으로 해결한다.

1. **원본을 원본 그대로 보존**한다
2. 정규화하여 **비교 가능한 형태**로 만든다
3. **다차원 점수**로 각 항목의 성향을 측정한다 (스킬인지, 에이전트인지, 워크플로인지...)
4. **verdict**(공식성, 신뢰성, 법적 상태, 활성도, 추천 상태)를 부여한다
5. 실제 사용을 위한 **curated pack**으로 재구성한다

## What DG Is

DG(Directive Graph)는 DKB 내부의 **다차원 해석 모델**이다.

각 아티팩트를 다차원 좌표의 한 점으로 표현한다. 이는 일반적인 3차원 그래프가 아니라, 정보 차원의 다차원으로 표현되는 형태다.

이렇게 표현된 점들의 군집을 통해:
- 사람들이 어떤 종류의 도구를 많이 만드는지
- 어떤 영역이 과밀하고 어떤 영역이 부족한지
- 중복/변종/보완 관계가 어떻게 형성되는지

를 통계적·분석적으로 파악할 수 있다.

## Core Principles

- **원본 불변**: 수집한 원본은 절대 수정하지 않는다
- **측정과 판정 분리**: 점수는 측정값이고, verdict는 정책 결과다
- **공식성과 신뢰성 분리**: 공식이라고 반드시 안전하지 않고, 비공식이라고 반드시 위험하지 않다
- **인기와 신뢰 분리**: stars/forks는 adoption 신호이지 trust 신호가 아니다
- **다차원 역할 표현**: "이것은 스킬이다"가 아니라 "skillness 0.81, agentness 0.44, workflowness 0.73"
- **로컬 우선**: 수집과 판정은 로컬에서 실행한다

## How It's Used

DKB는 개념 명세이며, 실제 운영은 구현체를 통해 이루어진다.

- **dkb-runtime**: DKB 명세를 구현한 설치 가능한 엔진
- **ai-store-dkb**: AI 생태계 전체를 리서치·수집하는 스토어 인스턴스
- **agent-prompt-dkb**: 에이전트 프롬프트를 큐레이션하는 인스턴스
- 다른 프로젝트(ClawVerse, Constellar 등)도 dkb-runtime을 설치해서 자체 DKB 인스턴스를 구축할 수 있다
